## 1. Framework and Execution

- **Framework**: [Vitest](https://vitest.dev/) (`^4.0.18`) is the sole testing framework for this project.
- **Location**: All test files are located in the `___tests___/` directory at the project root.
- **Naming**: Test files must follow the `*.test.ts` naming convention (e.g., `RunCodeReview.test.ts`).
- **Execution**: Run the full test suite with `npm run test`.

## 2. Test Strategy: Unit Tests First

The primary testing strategy is **unit testing** with a focus on isolation. Each class or module should be tested independently of its concrete dependencies.

- **Mocking**: Use Vitest's built-in mocking capabilities (`vi.fn()`, `vi.mock()`) to replace external dependencies.
  - When testing an application use case (e.g., `RunCodeReview`), all injected interfaces from the `core` layer (`IAiProvider`, `IFileScanner`, etc.) must be mocked. See `___tests___/ConfigurationAuditor.test.ts` for an example.
  - When testing an infrastructure adapter (e.g., `GeminiProvider`), mock external services like `fetch`. See `___tests___/GeminiProvider.test.ts`.
- **Focus**: Tests should validate the business logic within a single architectural layer, not the integration between layers.

## 3. What to Test

- **Business Logic**: Prioritize testing the `application` and `infrastructure` layers, as they contain the most complex logic.
- **Utility Functions**: All helper functions in `src/shared/utils/` should have comprehensive unit tests.
- **Error Conditions**: For every feature, include tests that verify the correct custom error (from `src/core/domain-errors/ReviewerErrors.ts`) is thrown under failure conditions. Use `expect(...).rejects.toThrow()` for async functions.
- **Edge Cases**: Test edge cases such as empty inputs, invalid data formats, and boundary conditions.

## 4. What NOT to Test

As configured in `vitest.config.ts`, we deliberately exclude certain files from coverage measurement to focus testing efforts:

- **Data Structures**: Simple data-only `entities` and `interfaces` in `src/core/` do not require tests.
- **Constants**: The `src/shared/constants.ts` file is excluded.
- **Entrypoints**: The main CLI entrypoint (`src/presentation/cli/main.ts`) and dependency injection container (`DependencyContainer.ts`) are difficult to unit test and are better covered by manual or end-to-end testing.

## 5. Code Coverage

- **Tooling**: Code coverage is generated by `v8` and can be viewed in the console or as an HTML report.
- **Thresholds**: The current coverage thresholds in `vitest.config.ts` are set low (`~35%`). This is a baseline. **All new code and bug fixes must include tests and aim to increase overall coverage, targeting at least 80% for the modified files.**
